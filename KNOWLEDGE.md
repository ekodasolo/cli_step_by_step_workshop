## CloudFormation と CLI の本質的な違い — 凝集 vs 疎結合

### CFn = 凝集（Cohesion）による一体管理

- スタック内のリソースは **ライフサイクルを共有** する
- 作成・更新・削除が「スタック単位」で一括で行われる
- `DependsOn` や `!Ref` でリソース同士が結合し、CFn がその依存関係を自動解決する
- **設計のポイント**: 同一のライフサイクルを共有するリソースをセンスよく凝集させてスタックにまとめると効果的
  - 例: ネットワーク（VPC/Subnet/IGW）とアプリケーション（EC2/ALB/ASG）はライフサイクルが異なるので、別スタックに分離する

### CLI = 疎結合（Loose Coupling）による個別管理

- リソース同士に暗黙の結合がない。ID を渡して自分で紐付ける
- 作成も削除も 1 リソースずつ。手間はかかるが **柔軟性がとても高い**
  - 1 つだけ差し替える、特定のリソースだけ残す、といった操作が自由にできる
- **ステートを持つリソースとの相性が良い**
  - データが入った DB、利用中の EIP、ログが蓄積された S3 バケットなど
  - CFn のスタック削除に巻き込みたくないリソースは、CLI で個別管理する方が安全

### ワークショップでの活用

この「凝集 vs 疎結合」の対比は、以下の場面で参加者に伝えると理解が深まる:

- **イントロダクション**: 「CFn が裏でやっていることを CLI で 1 つずつ体験する」の前提として
- **Step 09 クリーンアップ**: CFn ならスタック削除 1 回、CLI は 13 ステップの逆順削除。凝集の恩恵を最も実感する場面
- **まとめ**: どちらが優れているかではなく、特性を理解して使い分けることが重要

### CFn の真価 — 「型」としての再現性

CFn の再現性は、単に「手間を省く」ことではない。
**繰り返し用いられる鉄板構成を「型」として定義し、その型を再実行することに真価がある。**

- 一度きりの構成を CFn で書くのはオーバーエンジニアリング。CLI で十分
- 何度も作り直す環境（開発・検証・本番の横展開、ワークショップの環境準備など）にこそ CFn が活きる
- テンプレートは「この構成はこう作るべき」という組織の知見の結晶であり、コードレビューを経て洗練される
- つまり CFn テンプレートは「インフラの型紙」。型紙があるから、誰が作っても同じ品質の環境ができる

**判断基準**: 「この構成は今後も型として再実行されるか？」
- Yes → CFn / CDK で型を定義する
- No（一度きり、探索的、ステートフル）→ CLI で個別に扱う

### 実務での使い分け — 典型的な動機

明確な線引きは難しいが、「なぜそのツールを選ぶのか」の典型的な動機は整理できる。

#### CFn にしたい典型的な動機

| 動機 | 具体例 |
|------|--------|
| **同じ構成を何度も作る** | dev / stg / prod の横展開、ワークショップの環境準備 |
| **構成を型として組織に残したい** | 「Web 3 層の標準構成」をテンプレート化し、誰でも同じ品質で作れるようにする |
| **環境をまるごと作って、まるごと壊したい** | 検証環境を作っては壊す開発サイクル |
| **構成変更をレビュー可能にしたい** | テンプレートの diff をコードレビューし、変更の意図を記録に残す |
| **ドリフト（手動変更による乖離）を防ぎたい** | テンプレートが唯一の真実であり、手動変更を検知・是正できる |

#### CLI にしたい典型的な動機

| 動機 | 具体例 |
|------|--------|
| **ステートを持つリソースを守りたい** | データが入った DB、蓄積されたログの S3 バケットをスタック削除から切り離す |
| **一度きりの作業である** | 調査のための一時的なリソース作成、移行作業中の暫定構成 |
| **既存リソースをピンポイントで変更したい** | SG のルール 1 行だけ追加、タグだけ修正 |
| **状態を確認・調査したい** | `describe` でリソースの現在状態を確認、トラブルシュート |
| **探索的に構成を試したい** | 「まず手で作ってみて、動くことを確認してから CFn 化する」 |
| **CFn が対応していない操作がある** | 一部の API はCFn リソースとして未対応。CLI なら直接 API を叩ける |

#### 組み合わせのパターン

実務では「CFn か CLI か」の二者択一ではなく、組み合わせて使うことが多い。

- **骨格は CFn、微調整は CLI**: VPC + サブネット等の基盤は CFn で型化し、SG ルールの一時的な追加は CLI で
- **CFn で作って CLI で観察**: スタック作成後の状態確認やトラブルシュートは CLI の `describe` で
- **CLI で試して CFn に昇格**: 探索的に CLI で構成を検証し、固まったら CFn テンプレートに落とし込む

## CloudFormation でハマる典型的なパターン

CFn は強力なツールだが、原則を理解せずに使うとトラブルに陥る。
多くのトラブルは **2 つの根本原因** に集約される。

### 根本原因 1: Replacement の理解不足

CFn でリソースのプロパティを変更すると、変更内容によって 3 種類の挙動がある:

| 挙動 | 説明 |
|------|------|
| **No interruption** | リソースを停止せずにその場で更新 |
| **Some interruption** | 一時的な中断を伴う更新（再起動等） |
| **Replacement** | 新リソースを作成 → 旧リソースを削除（作り直し） |

**Replacement が危険な理由**:
- 旧リソースが **削除される**。データを持つリソース（DB、EBS 等）だとデータが消失する
- 物理名（`Name` や `DBInstanceIdentifier` 等）を直指定していると、同名リソースが既に存在するため新規作成が失敗する
- どのプロパティが Replacement を引き起こすかは、AWS ドキュメントの各リソースのページに記載されているが、把握しきれない

**対策**:
- 変更前に **Change Set** で「Replace かどうか」を必ず確認する
- データを持つリソースには `UpdateReplacePolicy: Retain` を設定する
- 物理名は可能な限り CFn に自動生成させる（直指定しない）

### 根本原因 2: ドリフト（手動変更による乖離）

CFn は **宣言的** にリソースを管理する。テンプレートが「あるべき姿」であり、更新時にはテンプレートの状態に収束させる。

**ドリフトが生じる典型的な場面**:
- 緊急対応でコンソールから SG ルールを手動追加した
- 運用中に CLI でタグやパラメータを直接変更した
- 他のスタックや自動化ツールが同じリソースを変更した

**ドリフトがあると何が起きるか**:
- 次のスタック更新で、手動変更が **上書き・消去** される
- テンプレートと現実の状態が乖離しているため、更新の影響が予測できない
- 最悪のケースでは `UPDATE_ROLLBACK_FAILED` に陥り、スタックが操作不能になる

**対策**:
- CFn 管理下のリソースは **手で触らない** を原則とする
- 変更したい場合は必ずテンプレートを修正してスタック更新する
- どうしても手動変更が必要な場合は、直後にテンプレートも同期させる
- 定期的に `detect-stack-drift` でドリフトを検出する

### データベースを CFn に入れるとハマりやすい理由

データベースはステートを持つリソースの代表格であり、上記 2 つの根本原因が最も深刻な影響を与える対象。

- **Replacement の被害が最大**: エンジンバージョン変更、ストレージタイプ変更などが Replacement を引き起こす場合があり、データごと消失するリスクがある
- **スタック削除で道連れになる**: アプリケーションスタックにDBを同居させると、スタック削除時にDBも消える。`DeletionPolicy: Retain` を入れ忘れると事故になる
- **ライフサイクルが異なる**: アプリは頻繁に作り直すが、DB のデータは永続。同じスタックに入れるべきではない

**これは「凝集 vs 疎結合」の議論そのもの**:
- DB はアプリとライフサイクルが違うのだから、同じスタックに凝集させるべきではない
- CFn で管理するとしても別スタックに分離し、`DeletionPolicy: Retain` と `UpdateReplacePolicy: Retain` を必ず設定する
- あるいは、DB は CFn の管理外に置いて CLI や手動で管理する選択肢もある

### まとめ: CFn のトラブルの大半はこの 2 つで説明できる

```
CFn は「テンプレートが正」の宣言的モデル
  │
  ├── Replacement を理解していない → 意図せずリソースが作り直される → データ消失
  │
  └── 手動変更でドリフトが生じる → テンプレートと現実が乖離 → 更新が予測不能に
        │
        └── 特にデータベース → ステートを持つので被害が最大
```

この原則を理解していれば、CFn のトラブルの大半は防げる。
逆に言えば、この原則を知らないまま CFn を使うのは危険であり、
CLI ワークショップで「API のナマの挙動」を体験しておくことの価値がここにある。
